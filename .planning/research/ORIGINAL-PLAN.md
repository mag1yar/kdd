# KDD — Substrate памяти и документации: план проекта

## Что это и чем не является

Плагин — это **слой состояния, памяти и документации**, независимый от исполнителя. Работу делает Claude Code (с Superpowers, GSD, любым плагином или без). Плагин отвечает ровно за одно: чтобы агент не забывал контекст, не изобретал заново уже решённое и не нарушал принятые решения и конвенции.

Чем не является: воркфлоу-движком, оркестратором, исполнителем. Ничего из того, что делают GSD / Superpowers / RuFlo, он не дублирует и с ними не конкурирует — он живёт *слоем ниже*. Исполнители быстро меняются и умирают, состояние переживать их не должно. Стабильный контракт между стором и исполнителем — **MCP**, а не формат файлов.

## Ключевой принцип: pull, не push

Плагин **не впрыскивает** контекст в каждый ход. Исполнитель (или его сабагент) запрашивает релевантное just-in-time через MCP. Профилактика загрязнения контекста (боль auto-memory RuFlo: тысячи почти-дублей). Качество держится на дисциплине извлечения и сборке контекст-пакета.

## Архитектура хранения (принятое решение)

| Стор | Что это | Где | Формат |
|---|---|---|---|
| **Доска / задачи** | статусы, приоритеты, колонки, ownership, комментарии | центрально, вне git, ключ = git-common-dir | SQLite (WAL) |
| **Решения / конвенции / спеки** | «как мы делаем», ADR, стиль слайсов | в репо, коммитится | Markdown |
| **Эпизодическая память** | что было в прошлых сессиях и почему | центрально, вне git | SQLite + вектор |
| **Заметки** | freeform-захват от ai/user | центрально, вне git | central-база + вектор |

Физически: одна центральная база (per-project, ключ = git-common-dir) + git-md решения в репо. База индексирует git-md, но git — канон.

### Почему доска не в git
- gitignored `.planning` → не переносится в новый worktree;
- коммитимая доска → мерж-конфликты между параллельными ветками.
Фикс: центральный стор, ключ = `git rev-parse --path-format=absolute --git-common-dir` — одинаков для всех worktree. Опц. снапшот доски в коммитимый md/json.

### RAG
- **must-not-miss** (активные конвенции для затронутых путей, обязывающие решения) — детерминированный retrieval по path/scope/tag, НЕ top-k.
- **nice-to-recall** (эпизоды, заметки) — RAG top-k через сабагент. RAG — v2.

## MCP-поверхность

Задачи: create_task, get_task, list_tasks, update_task, move_task, claim_task(worktree, branch), add_comment(author: ai|user), get_backlog.
Решения: record_decision, list_decisions, get_conventions(paths).
Заметки: create_note, list_notes, promote_note_to_decision.
Память: save_session_summary, search_memory, get_related_episodes.
Главное: **build_context_pack(task_id)** → задача + решения + конвенции для затронутых путей + top-k эпизодов. Retrieval policy — настоящий IP.

## Параллелизм и worktree
- Центральный стор → все worktree видят одну доску.
- claim_task фиксирует worktree+branch владельца → нет двойного захвата.
- SQLite WAL. «Фаза» и «спринт» — вьюхи/группировки над одним списком задач.

## Фазы
- **v0**: центральный sqlite для задач + коммитимый md для решений + тонкий MCP (get_task, update_task, record_decision, get_conventions). Без UI/вектора.
- **v0.5**: build_context_pack, claim_task, labels/priority.
- **v1**: канбан + бэклог как web-вьюха, комментарии/заметки, спринты как группировки.
- **v2**: эпизодическая память, вектор через сабагент, снапшот доски в git.

## Riски
- always-inject → загрязнение контекста. Держать pull.
- Слияние сторов по жизненному циклу → bloat.
- RAG на must-not-miss → молчаливые нарушения правил.
- Доска в git → конфликты + дыры в worktree.
- Скоуп-крип: ship v0 до UI/вектора.

## Дополнительные идеи пользователя (Kanban-driven команда)

1. **Автономный цикл**: Claude просыпается по cron/loop, проверяет канбан. Новая задача в New → агент-«бизнес-аналитик» проверяет содержимое, пишет комментарии/вопросы. Пользователь получает уведомление, отвечает. Агент снова просыпается, если ок — двигает в следующую колонку, где другой агент (планировщик) добавляет контекст: связанные коммиты, код (RAG).
2. **Per-task управление агентами**: на задаче задаётся модель и effort для агента; checkboxes — до какого этапа/колонки агенты могут брать задачу автономно (in analyze, in review, in progress...), дальше ведёт пользователь.
3. **Два режима работы плагина**: (а) просто документирование (человек ведёт всё сам), (б) полная работа с агентами.
4. **Задачи как память**: у агентов и оркестратора память = сами задачи; никто не забывает.
5. **Декомпозиция**: бизнес-аналитик разбивает задачу на child-задачи → параллельные агенты (фронт ждёт API от бэка, бэк разрабатывает; child-задачи в одном родителе = общая память).
6. **Спринты × worktree**: worktree 1 = спринт 1 (справочники), worktree 2 = спринт 2 (договор); параллельная разработка, merge после проверки. Без тегов/версий, просто n+1 спринтов.
7. **Тестировщик-агент**: через claude-in-chrome / playwright / agent-browser сам открывает вкладку и тестирует; map codebase в начале, карта обновляется при каждом изменении → агент строит тест-кейсы; оценки готовности.
8. **Анализ коммитов**: например, анализ коммитов бэка → автосоздание задач на фронт.
9. **UI**: пользователь через web-UI добавляет/редактирует задачи как в Jira/Azure DevOps (связанные задачи, приоритет, оценка); Claude — через MCP.

## Контекст боли пользователя
GSD крут вначале, но устаёшь: агенты пишут отсебятину. Его `.planning` папка идеальна для документации/бэклогов/фаз/milestones. Работа в нескольких worktree; `.planning` в gitignore → сложности с worktree. Superpowers — реже отсебятина, быстрее, но тоже бывает. Главная цель: документировать задачи и спринты, ничего не забывать, всё хранится и видно.
